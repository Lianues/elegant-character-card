# message 文件命名与顺序说明

本文说明 `message/` 目录中文件名（`1.md`、`2.md`、`100.md`）的含义，以及在 `repo/build` 流程中的构建行为。

---

## 1. message 是什么

在当前项目中：

- `data.message`：角色的消息数组（对齐 `character.get` 风格）
- 用于存放完整问候序列（首条消息 + 其余可选消息）

> 当前已统一：首条消息也放在 `message[0]`，不再单独使用 `first_mes`。

---

## 2. 文件名的含义

`message/` 目录中的文件名采用：

- `0.md`
- `1.md`
- `2.md`
- ...
- `99.md`

规则：

- 文件名里的数字就是该消息在 `message` 数组里的索引（0-based）
- **不再补零**（不是 `001.md`）

对应关系示例：

- `0.md` -> `message[0]`
- `1.md` -> `message[1]`
- `10.md` -> `message[10]`

---

## 3. 构建（build）时的顺序规则

当执行 `build` 从仓库重建角色卡时：

1. 会读取 `message/` 目录中的 `.md` 文件
2. 按文件名排序后写入 `data.message`

当前排序策略：

- **只允许纯数字文件名**（`^\d+\.md$`）
- 文件名按**数值升序**排序（`0,1,2,3,...,10,11,...`）

如果出现非数字文件名，构建会直接报错。数字文件名不要求从 0 开始，也不要求连续。
因此 `1.md`、`2.md`、`10.md` 会按预期顺序构建，不会出现字符串排序导致的 `1,10,2` 问题。

---

## 4. repo（拆分）时会如何生成

当执行 `repo` 把 JSON/PNG 拆分为仓库时：

- `data.message[0]`（首条消息）写到 `message/0.md`
- `data.message[1]` 写到 `message/1.md`
- 以此类推

---

## 5. 建议

- 尽量只使用纯数字文件名（`1.md`、`2.md`）来避免歧义
- 如果你手动新增消息，请按连续数字命名
- 删除中间编号会改变最终数组长度与顺序，建议同步整理
