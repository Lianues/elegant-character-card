# 01 · 仓库结构总览

## 1. 完整目录布局

```
my_character/                         ← 仓库根目录（与角色名同名，非法字符会被替换为 _）
├── _metadata.yaml                    ← 主元数据（spec / spec_version / data 各标量字段 + 路径标记）
├── character.png                     ← 默认底图（可选，由 image_path 指向）
├── docs/                             ← 本套 yaml 规范文档（自动从项目复制）
├── message/                          ← 角色消息（首条 + 备选问候）
│   └── 0.md, 1.md, ...
├── extensions/                       ← 角色级 SillyTavern 扩展
│   └── _metadata.yaml
├── world_book/                       ← 世界书
│   ├── _metadata.yaml                ← 世界书自身的元数据
│   └── entries/                      ← 每条世界书条目
│       ├── 0_alice.yaml
│       ├── 0_alice.md
│       └── ...
└── group_only_greetings/             ← 仅群聊问候（结构同 message/）
```

> 子目录的具体字段规范分别看 [`02_extensions目录.md`](./02_extensions目录.md)、[`03_message目录.md`](./03_message目录.md)、[`04_world_book条目.md`](./04_world_book条目.md)。

## 2. 主 `_metadata.yaml`

放在仓库根目录，对应酒馆角色卡的最外层。示例：

```yaml
spec: chara_card_v3
spec_version: "3.0"
data:
  name: 示例角色卡
  tags: []
  creator: ""
  character_version: ""
  description: ""
  personality: ""
  scenario: ""
  system_prompt: ""
  post_history_instructions: ""
  mes_example: ""
  creator_notes: ""
  message: my_character\message                       # 路径标记（指向同名子目录）
  group_only_greetings: my_character\group_only_greetings
  extensions: my_character\extensions
  world_book: my_character\world_book
  image_path: character.png                           # 默认底图（项目内部字段）
```

### 2.1 字段规则

| 字段 | 类型 | 在哪里编辑 |
|---|---|---|
| `spec` / `spec_version` | 字符串 | 一般不动，固定 `chara_card_v3` / `"3.0"` |
| `name / description / personality / scenario / mes_example / creator_notes / system_prompt / post_history_instructions / creator / character_version / tags` | 标量/数组 | **直接在主 `_metadata.yaml` 里编辑** |
| `message / group_only_greetings / extensions / world_book` | 字符串（路径标记） | **不要改成实际值**；路径只是标记符，真实数据在对应子目录 |
| `image_path` | 字符串 | 编辑这里改默认底图，详见 §3 |

### 2.2 路径标记机制

像 `message: my_character\message` 这种"看起来是文件路径"的字段，**不是真实数据**——它是仓库化机制的"标记符"，告诉 build 命令"这个字段的真实值在 `<repo>/message/` 子目录里"。

- 如果你**不想让某字段被分离**，把它从子目录的 yaml 中删除，把真实值（字符串/数组）写在主 `_metadata.yaml` 里即可。
- 如果你**新增了某字段对应的子目录**（例如手工建了 `message/` 但没在主 yaml 里写路径标记），build 时会忽略该子目录。

### 2.3 已自动剔除的"原版不存在"字段

以下 V3 schema 默认值会被 `repo` 自动从 YAML 删除（保持仓库简洁）；它们在 build 时由 schema 自动用 `null` / `[]` 填回：

```
nickname, source, creation_date, modification_date,
assets, creator_notes_multilingual
```

如果你确实需要这些字段，直接在 `_metadata.yaml` 的 `data:` 下手动加上即可。

## 3. `image_path` —— `build -f png` 的默认底图

```yaml
data:
  ...
  image_path: character.png    # 相对路径（基于仓库根目录）或绝对路径
```

### 3.1 `repo` 时的自动行为

| 输入 | `image_path` 写入值 | 同时操作 |
|---|---|---|
| PNG 角色卡 | `character.png` | 自动剥离 `ccv3` / `chara` 元数据 chunk，裸图保存为 `<repo>/character.png` |
| JSON 角色卡 | `""`（空字符串） | 不生成图 |

### 3.2 `build -f png` 时的解析优先级

| 优先级 | 来源 | 文件不存在时 |
|---|---|---|
| 1 | 命令行 `-b <file>` | ❌ **报错** |
| 2 | `data.image_path` 已填值 | ❌ **报错**（不静默回落，避免用户改了路径却不知道） |
| 3 | `data.image_path` 为空字符串 | ✅ 自动使用项目内置 `default.png` |

> 设计逻辑：**只有"明确没指定底图"时才用默认图；一旦指定了底图但缺失，就一定会报错**。

### 3.3 `image_path` 不会出现在导出的角色卡里

它是项目内部字段，反向构建时被 V3 schema 自动忽略，不会污染 SillyTavern 导出。

## 4. 相关命令速查

> CLI 命令名为 **`ecc`**（短）或 `elegant-character-card`（完整）。未安装时也可用 `npx ecc <args>`。

```bash
# 仓库化（PNG 输入：自动剥离 chunk + 写 image_path: character.png）
ecc repo character.png

# 仓库化（JSON 输入：image_path 留空）
ecc repo character.json

# 重建为 SillyTavern 兼容 JSON
ecc build my_character -f json

# 重建为 PNG（使用 image_path 指定的默认底图）
ecc build my_character -f png

# 重建为 PNG，覆盖默认底图
ecc build my_character -f png -b cover.png
```
